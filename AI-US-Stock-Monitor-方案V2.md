# AI US Stock Monitor —— Agent 实操方案 V2

## 一、需求结构化梳理（确认理解）

**核心目标**：把你每天手动刷 X 博主、自己琢磨大盘涨跌原因的时间，转化为一份自动生成的盘后日报，降低信息搜集成本，辅助发现投资机会和理解行业变化。

**两大功能模块**：

模块一｜热股资讯追踪——追踪指定 X 博主在目标交易日相关时间窗口内的发帖，提炼热门个股、AI 产业链热点、其他值得留意的信息。

模块二｜美股大盘复盘——分析最近一个已收盘美股交易日的大盘涨跌归因（政策/地缘/财报/超级个股/领导人言论/其他）。

**关注范围**：AI 上下游、互联网科技股、纳斯达克指数。

**触发与输出**：每日北京时间 06:30 生成最近一个已收盘美股交易日的日报。

> 时间口径说明：不再按中国自然日 00:00-23:59 生成日报，而是以“最近一个完整美股交易日”为核心。北京时间 06:30 通常已经覆盖美股常规交易收盘、盘后初步新闻和主要财经媒体归因，适合作为个人盘后复盘时间点。

## 二、必须先解决的关键风险

这些问题不解决，后面的架构设计都是空中楼阁，建议先用小样本验证清楚。

### 风险一：X 博主数据怎么拿，这是整个项目的生死线

X（推特）的数据获取在 2023 年之后变得非常受限，不能把“稳定抓取 X”当成天然成立的前提。这个项目的数据源建议按三层设计：

#### 第一层：开源低成本 MVP 数据源

优先验证 `twscrape`。它是一个 Python 异步库和 CLI，支持通过登录态访问 X/Twitter Search、用户信息、用户发帖、列表时间线等接口，适合个人项目先验证“指定博主每日动态 → 热点提炼 → 日报生成”的闭环。

但需要明确：`twscrape` 依赖 X 网页/GraphQL 接口和账号登录态，存在被限流、账号风控、接口变化导致失效的风险。因此它适合作为 MVP 和个人研究工具，不应承诺为长期稳定生产方案。

#### 第二层：兜底数据源

当 X 抓取失败时，系统不应直接中断日报，而是降级使用：

- `web_search` 搜索指定博主名、股票代码、AI 产业链关键词和日期
- 财经新闻源补充大盘和行业事件
- 可选 RSSHub/Nitter 类方案作为补充，但不作为唯一主数据源

兜底日报必须明确标注“今日 X 原始抓取不完整”，避免让用户误以为数据覆盖完整。

#### 第三层：稳定化升级路径

如果 MVP 被证明有价值，再考虑升级到：

- X 官方 API：稳定性和合规性更好，但需要开发者账号和付费额度
- 第三方托管抓取服务：上线快，但有持续费用和供应商依赖

第一版默认选择开源低成本路线：`twscrape` 主抓取，`web_search`/财经新闻兜底，官方 API 暂不接入。

### 风险二：你提到的“博主名单文档”还没有落到项目配置里

你说“具体追踪哪些博主见项目 X 博主文档详情”，但当前项目目录里还没有结构化的博主名单配置。这份名单非常关键，因为它会影响：

- 这些博主的活跃度和发帖语言（中文还是英文，直接决定关键词、Prompt 和翻译策略）
- 是否经常发 replies/retweets，决定是否默认排除互动噪音
- 是否是公开内容，还是需要登录后才能看到
- 名单数量（5 个博主和 50 个博主，抓取频率、限流风险、落库设计完全不同）

建议第一版把博主名单维护为 `accounts.yaml`，后续日报生成直接读取这份配置。

### 风险三：行情数据源还需要确定

模块二的大盘复盘需要稳定拿到纳斯达克、标普 500、道琼斯等指数的收盘数据。可选路线：

- Alpha Vantage：接口清晰，但免费额度有限
- Yahoo Finance：个人项目常用，但非正式 API 稳定性一般
- Finnhub：接口体验较好，但需要注册 API Key
- 新闻搜索兜底：最快验证流程，但行情数字和归因结构不够稳定

MVP 阶段建议先用新闻搜索 + 一个免费行情源验证，后续再固定行情数据供应商。

### 风险四：日报投递方式与节假日策略还未定

当前文档只定义“输出日报”，还没有决定投递形态。MVP 建议先输出 Markdown 文件，避免过早接入邮件、飞书、Notion 等外部服务。

美股休市日也需要明确策略：第一版建议生成一份简短的“无交易日报”，说明休市原因，并只保留重大新闻和 X 热点补充。

## 三、整体架构

```
触发：每日北京时间 06:30
   ↓
确定最近一个已收盘美股交易日
   ↓
模块一：热股资讯追踪 ──┐
模块二：美股大盘复盘 ──┤
   ↓                  │
   └──→ 整合 Agent ←───┘
   ↓
输出：统一格式盘后日报
```

两个模块并行执行、各自走“执行者 → 验证者”的循环，最后由一个整合 Agent 合并成最终日报。这样设计的好处是：两个模块的判断标准完全不同（一个是社交媒体舆情归因，一个是宏观/市场归因），分开设计验证标准会更精确，不会互相干扰。

## 四、模块一详细设计：热股资讯追踪

### 数据采集（MVP 阶段）

第一版不再只依赖 `web_search`，而是采用“指定账号抓取 + 关键词补充 + 失败兜底”的结构。

#### 1. 输入配置

维护一份 `accounts.yaml` 或等价配置文件：

```yaml
accounts:
  - handle: example_user
    display_name: 示例博主
    group: AI/科技股
    language: zh
    weight: 1.0
    include_replies: false
    include_retweets: false
```

字段含义：

- `handle`：X 用户名
- `display_name`：日报中展示的博主名称
- `group`：博主分组，如 AI、科技股、宏观、美股交易
- `language`：主要语言，用于后续搜索和摘要策略
- `weight`：作者权重，后续用于热点排序
- `include_replies`：默认不抓 replies，避免噪音过大
- `include_retweets`：默认不抓 retweets，除非该博主经常通过转发表达观点

#### 2. 主采集方式：twscrape

使用 `twscrape` 获取指定博主的公开 posts：

- 启动时读取 X 登录 cookie 或本地账号池
- 按 handle 查询 user id
- 对每个 user id 拉取 `user_tweets`
- 按目标时间窗口过滤 posts
- 保存原始 post 到本地 SQLite
- 使用 post id 去重，避免重复分析

建议保存字段：

- `post_id`
- `author_handle`
- `author_name`
- `created_at`
- `text`
- `lang`
- `url`
- `reply_count`
- `repost_count`
- `like_count`
- `view_count`（如果可用）
- `raw_json`
- `fetched_at`

#### 3. 时间窗口

模块一的 X 博主动态应服务于“最近一个已收盘美股交易日”的盘后日报，而不是简单按中国自然日切割。

建议第一版使用：

- 起点：上一个报告生成时间之后，或目标交易日前一日美股盘后开始
- 终点：北京时间当日 06:30 生成日报时刻
- 存储：统一转换为 UTC
- 展示：日报中按北京时间展示，涉及美股事件时注明对应交易日

这样可以覆盖：

- 美股盘前/盘中/盘后的博主讨论
- 收盘后的初步复盘
- 盘后财报、重大公告和新闻反应

#### 4. 关键词补充

仅靠固定博主可能会漏掉全网热点，因此增加关键词补充搜索：

- 股票代码：`NVDA OR AMD OR AVGO OR TSM OR MSFT OR GOOGL OR META`
- AI 产业链：`AI chips`, `data center`, `HBM`, `optical`, `power`, `cooling`
- 中文关键词：`美股 AI 算力 芯片 存储 光通信 电力`
- 热点搜索：`美股 热门股票 今日`, `Nasdaq rally reason today`

关键词补充数据只作为“候选热点”，不能覆盖掉指定博主数据。日报中应区分“监测博主提及”和“外部搜索补充”。

#### 5. 抓取失败处理

如果某个账号抓取失败：

- 记录失败原因：限流、登录失效、账号不存在、网络错误、解析错误
- 不重试过多次，避免触发风控
- 在日报数据来源说明中标注未成功覆盖的账号数量
- 触发 `web_search` 兜底，搜索该博主 handle 与当日关键词

如果全部 X 抓取失败：

- 仍生成日报
- 明确标注“今日 X 博主原始数据不可用，本报告基于搜索和新闻兜底生成”
- 热门个股部分降低置信度，不输出过度确定结论

### 抓取数据处理流程

```
读取博主配置
   ↓
twscrape 拉取指定账号 posts
   ↓
按时间窗口过滤
   ↓
SQLite 原始数据落库
   ↓
去重与清洗
   ↓
抽取 ticker / 公司名 / 行业主题 / 事件关键词
   ↓
按作者权重、提及次数、互动量计算热度
   ↓
交给 Executor Agent 生成日报
   ↓
Verifier Agent 检查结论是否能追溯到原始 post
```

清洗规则：

- 去除重复 post
- 默认排除纯转发、抽奖、无实质观点的互动内容
- 保留原文，不在采集阶段做过度摘要
- 对长帖、thread、引用帖保留上下文链接
- 中英文内容都进入分析，但日报统一输出中文

热度排序建议：

- `提及次数`：同一股票/主题被多少条 post 提及
- `作者覆盖`：是否被多个不同博主提及
- `作者权重`：高质量博主权重更高
- `互动强度`：likes、reposts、replies、views 作为辅助
- `新鲜度`：越接近日报生成时间，权重略高

### 知识文件（Skill）——分类标准必须写清楚

```markdown
# 热股资讯分类标准

## 分类一：热门个股
判断标准：目标交易日前后被多个监测博主提及，或讨论热度明显异常的个股。
必须说明“为什么”，归类到以下原因之一（或多个）：
- 风口/赛道叙事（如沾上 AI 概念）
- 重大订单/合作落地
- 上下游关键企业动态影响
- 估值严重偏离讨论（被低估/高估）
- 政策或资源利好
- 财报、指引、管理层言论影响

## 分类二：AI 产业链热点
细分领域：算力、芯片、存储、光通信、散热、电力等 AI 基础设施。
判断标准：目标交易日该领域被反复提及的具体事件，不是泛泛的“AI 很热”这种空话。

## 分类三：AI 大模型公司动态
判断标准：模型公司（OpenAI、Google、Anthropic、Meta 等）的产品发布、战略调整、人事变动等可能影响股价的具体事件。

## 分类四：其他值得留意
判断标准：可能影响行业格局或股市情绪的非个股类信息，如行业泡沫论调、技术路线颠覆性讨论。
注意：这一类容易变成“什么都往里塞”，必须有明确的“为什么值得留意”的理由，不能只是凑数。
```

```
你是一名美股社交媒体舆情分析助手。基于以下 X 博主 posts、搜索补充结果和分类标准，生成目标交易日的热股资讯简报。

分类标准：
{插入上方 Skill 文件内容}

监测博主名单：
{插入 accounts.yaml 中的博主配置}

X 原始抓取结果：
{插入 twscrape 获取并清洗后的 posts}

搜索/新闻补充结果：
{插入 web_search 或财经新闻补充结果}

数据完整性：
{成功抓取账号数 / 总账号数、失败账号列表、是否使用兜底数据}

要求：
1. 每条结论必须能追溯到具体 post 或搜索/新闻来源，不能凭空推断。
2. 如果某个分类今天没有明显内容，直接说明“今日该类别无显著动态”，不要硬凑。
3. 每条热点用一句话说清楚“是什么 + 为什么”，避免空泛描述。
4. 明确标注信息来源：优先标注具体博主；如果来自兜底搜索，则标注新闻或搜索来源。
5. 区分“监测博主共同提及”和“外部搜索补充发现”，不要把两者混为一谈。
```

### 验证者 Agent（Verifier）

```
5. 是否区分了 X 原始抓取和兜底搜索来源。
6. 是否存在原始数据里有但简报漏掉的明显热点。

待审核简报：
{插入执行者输出}

原始数据：
{插入 X posts、搜索补充结果、数据完整性说明}

输出格式：

- 具体问题列表（如不合格）
```

## 五、模块二详细设计：美股大盘复盘

### 数据采集

模块二只分析最近一个已收盘美股交易日，不按中国自然日切割。

需要采集：

- 大盘指数收盘数据：纳斯达克、标普 500、道琼斯涨跌幅
- 关键科技股表现：NVDA、MSFT、GOOGL、META、AMZN、AMD、AVGO 等
- 新闻搜索，分别针对归因方向：
  - `美联储 利率预期 美股 {交易日}`
  - `地缘政治 美股影响 {交易日}`
  - `财报 美股 大盘 {交易日}`
  - `Nasdaq rally reason {交易日}`
  - `Nasdaq selloff reason {交易日}`
  - `Trump comments market {交易日}`

MVP 阶段可以先用财经新闻搜索和免费行情源，等流程稳定后再固定为 Alpha Vantage、Yahoo Finance 或 Finnhub。

### 执行者 Agent（Executor）

```
你是一名美股宏观分析助手，任务是解释最近一个已收盘美股交易日的大盘涨跌核心原因。

请按以下维度逐一核查并归因（如某维度今天无明显影响，直接说明“今日无明显影响”）：
1. 政策：货币政策（降息/加息预期）、监管政策（对 AI 等行业的利好或限制）
2. 地缘：地缘政治事件对市场情绪的影响
3. 财报：是否有行业龙头财报带动大盘整体情绪
4. 超级个股：是否有资金过度集中在某个体量极大的个股或新股 IPO，导致市场资金分布失衡
5. 领导人言论：政治人物言论引发的行业或大盘波动
6. 其他：不属于以上五类但确实造成影响的因素

目标交易日：
{最近一个已收盘美股交易日}

大盘数据：
{插入纳斯达克/标普/道琼斯收盘数据}

新闻搜索结果：
{插入搜索结果}

要求：
1. 每个归因都要有新闻或数据依据支撑，不能主观臆测。
2. 多个因素同时存在时，按影响程度排序呈现，不要平均分配篇幅。
3. 用普通投资者能看懂的语言，避免堆砌专业术语。
4. 避免过度确定的表达，把“可能、市场普遍认为、媒体归因”区分清楚。
```

### 验证者 Agent（Verifier）

```
你是内容审核员，检查大盘复盘分析是否合格。

逐条检查：
1. 每个归因是否有新闻数据支撑，还是分析师自己的主观判断。
2. 是否存在“听起来合理但查不到依据”的内容。
3. 排序是否真的反映了影响程度，还是平铺直叙没有侧重。
4. 语言是否存在过度确定的表述（比如把“可能”说成“一定”）。
5. 是否使用了最近一个已收盘美股交易日，而不是中国自然日。

待审核内容：
{插入执行者输出}

原始数据与新闻：
{插入原始数据}

输出格式同模块一。
```

## 七、Memory 设计（防止内容重复、追踪趋势）

建议本地存一个 `history.json`，记录最近 5-7 份报告中出现过的个股和主题。这样执行者 Agent 生成新一天的报告时，可以参考历史记录，做出“这是某股票连续第 3 天被讨论”这类趋势性判断，而不是每天都是孤立的快照。

建议记录字段：

- `report_date`
- `tickers`
- `topics`
- `source_count`
- `mentioned_authors`
- `summary`
- `confidence`

这个设计对实际决策价值更高：连续被讨论的热点，往往比单日突发的热点更值得关注。

---

## 八、建议的实施顺序

不要一次性把两个模块都做，按这个顺序推进：

**第一步**：验证 `twscrape` 对 3-5 个公开账号是否可用。确认能拿到 posts、能按时间过滤、能落库去重、能识别失败原因。

**第二步**：做模块二（大盘复盘）。因为数据源更标准化（新闻搜索 + 指数数据），更容易验证“执行者 + 验证者”这套机制本身好不好用。

**第三步**：做模块一的完整 MVP。把 `twscrape` 原始抓取、关键词补充、失败兜底、热点提炼串起来。

**第四步**：两个模块都验证稳定后，做整合 Agent 和定时触发，正式跑成每日北京时间 06:30 的盘后日报。

---

## 九、需要你确认或提供的信息

除具体博主名单外，还有这些问题需要在实施前讨论清楚：

1. **X 登录态**：`twscrape` 需要 X 登录 cookie 或账号池；这部分无法完全“无账号”稳定抓取。需要你决定是否接受本地保存登录 cookie。
2. **博主名单配置**：除了 handle，还需要知道每个博主的语言、分组、权重、是否包含 replies/retweets。
3. **数据合规边界**：第一版建议只抓公开内容，不抓私密账号、不绕过访问限制。
4. **行情数据源**：需要确定用 Alpha Vantage、Yahoo Finance、Finnhub，还是先用新闻搜索和免费行情源。
5. **日报投递方式**：MVP 建议先输出 Markdown 文件；邮件、飞书/企微、Notion、本地网页可以后续再接。
6. **节假日处理**：建议美股休市日生成“无交易日报”，说明休市原因，并保留重大新闻和 X 热点补充。

## 十、测试与验收标准

MVP 的验证重点不是“日报看起来很漂亮”，而是确认整条链路能可靠跑通：

- 用 3-5 个公开 X 账号试跑 `twscrape`，确认能拿到 posts。
- 连续运行两次，确认 post id 去重有效。
- 模拟单账号抓取失败，确认能降级到搜索兜底。
- 验证北京时间 06:30 时能正确定位“最近一个已收盘美股交易日”。
- 抽查日报每条热点是否能追溯到原始 post 或新闻来源。
- 验证全部 X 抓取失败时，日报仍能生成，并明确标注“数据不完整”。
